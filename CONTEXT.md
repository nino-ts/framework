- "nino.ts", "ninots", "ninoTS" é um projeto de código fonte aberto https://github.com/nino-ts
- Pertence a "pandowLABS" https://github.com/pandowlabs
- Criado por mim: https://github.com/joaovjo
- Atualmente desenvolvido por mim João Vitor de Jesus Oliveira (https://joaovjo.com.br, https://github.com/joaovjo) e pelo Victor Geruso (https://geruso.com, https://github.com/vgeruso)
- "ninoTS" é escrito em TypeScript e Bun native, inicialmente sem suporte a outros runtimes
- "ninoTS" não possui dependências externas, seu código é escrito inteiro do zero, usando APENAS as APIs nativas do Bun
- "ninoTS" não reescreve a roda, se um recurso já existe no Bun, ele usa esse recurso e às vezes faz um wrapper para simplificar a utilização
- "ninoTS" é um framework fullstack baseado na Developer Experience do Laravel e do Next.js, tentando absorver ao máximo os benefícios de ambos ecossistemas, mas NÃO POSSUI compatibilidade com ambos
- No GitHub nino-ts é a organização, nino-ts/framework é o meta-package e nino-ts/ninots é o starter kit, ou seja, o template de projeto para começar a usar o framework
- O framework é um monorepo, ou seja, tem vários pacotes dentro do mesmo repositório, cada um com sua função específica, como por exemplo: nino-ts/foundation, nino-ts/http, nino-ts/router, nino-ts/orm, etc
- O framework é modular, ou seja, cada pacote pode ser usado de forma independente, sem precisar instalar o framework inteiro, por exemplo, você pode usar apenas o nino-ts/http para criar um servidor HTTP, ou apenas o nino-ts/orm para usar o ORM, etc
- O ninots segue essa separação entre o starter kit e o framework baseando-se no mesmo formato do Laravel: https://github.com/laravel/laravel é o starter kit do Laravel, enquanto o https://github.com/laravel/framework é o framework base, ou seja, a implementação dos recursos do Laravel
- No ninots, o código do framework é importado como uma dependência, ou seja, você instala o nino-ts/framework e importa os recursos que precisa, por exemplo: import { createServer } from 'nino-ts/framework/http'
- O framework possui um src que é onde está a "cola" do framework, ou seja, a implementação dos recursos dos pacotes internos já que nenhum pacote interno importa outro pacote interno, ou seja, o nino-ts/http não importa o nino-ts/foundation, o nino-ts/orm não importa o nino-ts/router, etc, isso para evitar dependências cíclicas e manter a modularidade do framework
- O framework é escrito em TypeScript estrito e tem uma tipagem forte, ou seja, todas as funções, classes, interfaces, etc, possuem tipos definidos e o código é verificado pelo TypeScript para garantir a segurança de tipos
- O framework é testado usando o Bun test runner, ou seja, os testes são escritos usando a API de testes do Bun e são executados usando o comando bun test, garantindo que o código esteja funcionando corretamente e que as mudanças não quebrem funcionalidades existentes
- TDD é a abordagem de desenvolvimento utilizada no framework, ou seja, os testes são escritos antes do código de implementação, garantindo que o código seja testável e que as funcionalidades sejam implementadas de forma correta desde o início
- O framework é documentado usando o TypeDoc, ou seja, a documentação é gerada a partir dos comentários de código usando a sintaxe do TSDoc, garantindo que a documentação esteja sempre atualizada e que os usuários possam entender como usar os recursos do framework de forma fácil e rápida
- O framework é licenciado sob a licença MIT, ou seja, é um software de código aberto que pode ser usado, modificado e distribuído por qualquer pessoa, desde que a licença seja incluída em todas as cópias ou partes substanciais do software
- O framework é mantido por uma comunidade de desenvolvedores, ou seja, qualquer pessoa pode contribuir para o desenvolvimento do framework, seja corrigindo bugs, implementando novos recursos, escrevendo testes, melhorando a documentação, etc, e as contribuições são revisadas e aceitas pela equipe de manutenção do framework para garantir a qualidade do código e a consistência do projeto
- O framework é atualizado regularmente, ou seja, novas versões são lançadas com correções de bugs, melhorias de desempenho, novos recursos, etc, garantindo que o framework esteja sempre evoluindo e atendendo às necessidades dos usuários e da comunidade de desenvolvedores
- O framework é focado em ser leve e rápido, ou seja, o código é otimimizado para ter um desempenho excelente, utilizando as melhores práticas de desenvolvimento e as APIs nativas do Bun para garantir que as aplicações construídas com o framework sejam rápidas e eficientes, mesmo em cenários de alta carga e tráfego intenso
- O framework é projetado para ser fácil de usar e aprender, ou seja, a API do framework é intuitiva e consistente, seguindo as melhores práticas de design de APIs e a Developer Experience do Laravel e do Next.js, garantindo que os desenvolvedores possam começar a usar o framework rapidamente e sem dificuldades, mesmo que sejam iniciantes em desenvolvimento web ou em TypeScript
- O framework é flexível e personalizável, ou seja, os desenvolvedores podem configurar e estender o framework de acordo com suas necessidades, seja adicionando novos recursos, modificando o comportamento padrão, ou integrando com outras bibliotecas e ferramentas, garantindo que o framework possa ser adaptado para atender às necessidades específicas de cada projeto e equipe de desenvolvimento
- O framework é seguro, ou seja, possui recursos e práticas de segurança integrados para proteger as aplicações construídas com ele contra vulnerabilidades comuns, como injeção de SQL, cross-site scripting (XSS), cross-site request forgery (CSRF), etc, garantindo que as aplicações sejam protegidas contra ataques e que os dados dos usuários estejam seguros
- O framework é escalável, ou seja, pode ser usado para construir aplicações de qualquer tamanho e complexidade, desde pequenos projetos pessoais até grandes aplicações empresariais, garantindo que o framework possa crescer junto com as necessidades dos desenvolvedores e das equipes de desenvolvimento, e que possa ser usado para construir aplicações robustas e de alta qualidade em qualquer cenário
- O ninots segue uma arquitetura híbrida modular, os modulos da aplicação podem ser escritos uma única vez e copiados para outros projetos, ou seja, os modulos não possuem dependências entre si, garantindo que eles possam ser reutilizados em outros projetos sem precisar de ajustes ou modificações, e que possam ser compartilhados entre equipes de desenvolvimento para acelerar o processo de desenvolvimento e promover a colaboração entre desenvolvedores
